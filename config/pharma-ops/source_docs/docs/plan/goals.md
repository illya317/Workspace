# Pharma Ops 电子记录 — 目标

> 从 `复刻计划.md` 拆分，定义总目标和总体原则。

## 总目标

把原始 `raw/*.docx` / `schema/md_canonical/*.md` 中约 16 个产品的批检验记录，改造成可保存、可回填、可自动计算、可自动判定的电子表单系统。

本计划的核心不是"多做几个页面"，而是先建立一条可验证的信息化闭环：

1. 从 MD 真源稿中识别产品、阶段、检测项、字段、公式、判定和结论。
2. 用 YAML 表达细则和字段，不在 YAML 中写 HTML 表格或 `<input>`。
3. 后端输出统一结构 JSON。
4. 前端按 JSON 动态渲染表格、输入、只读计算项和结论。
5. 所有填写值、日期、勾选、计算结果、结论都能保存并刷新回填。
6. 已用别嘌醇片建立完整纵切样板，后续按同一验收闭环推广到 16 个产品。

## 总体原则

### 数据源优先级

修改检测项内容时按以下优先级：

1. 用户明确给出的参考图或文字说明。
2. `schema/md_canonical/<品种>.md` 中的真源稿。
3. 现有 YAML 配置。
4. Agent 的推断。

Agent 推断出的内容必须标记为 `needs_review: true` 或进入缺口报告，不得伪装成已确认。

### YAML 分工

`config/products.yaml`：

- 只做产品索引和品种到模板文件的映射。
- 不承载复杂字段细则。
- 可保留旧方法映射作为迁移参考，但最终应收敛为产品索引。

`config/record_templates/*.yaml`：

- 品种级记录结构。
- 包含阶段、检验前确认、检测项列表、操作方法纯文案、标准规定、清场、结论配置、品种参数。
- 不允许包含 `<table>`、`<input>`、`<select>`、`<br>` 等 HTML。

`config/methods/*.yaml`：

- 方法级字段定义。
- 包含字段名、类型、单位、默认值、公式、判定规则、repeat、组件 `render_as`。
- 所有自动计算和自动审核所需字段必须在这里或检测项 `config` 中表达。

### 字段作用域

每个可填写或可计算字段必须有稳定作用域：

| scope | 用途 | 示例 |
|---|---|---|
| `global` | 整个系统或所有批次复用 | 检验者姓名、复核者姓名 |
| `batch` | 当前批次所有页面复用 | 批号、产品、批次状态 |
| `stage` | 当前阶段复用 | 请验日期、检验日期、报告日期、检品数量 |
| `test` | 当前检测项内有效 | 水分1、水分2、样1-OD |
| `repeat` | repeat 行或列 | 片1-重、样品3-OD |

字段 key 必须稳定，不能依赖 DOM id 或中文位置猜测。建议格式：

```text
batch.batch_number
stage.intermediate.inspect_date
test.intermediate.content.sample_1_od
test.finished.dissolution.sample_6_result
```

如果继续使用路径式 key，也必须在后端结构中输出 `field_key`，前端和数据库统一使用它。

### 前端渲染原则

- 页面框架可以用 Jinja，但具体数据表格和字段由 JS 根据结构渲染。
- 通用组件接收字段结构和配置，不写死品种名、常数、波长、规格。
- 计算项 `attr: calculated` 必须只读。
- 预填项 `attr: prefilled` 可以显示默认值，是否可编辑由字段配置决定。
- 用户人工覆盖自动结论时，需要有 override 状态，避免下次计算静默改写。

### 验收原则

每个阶段完成后，Agent 必须提交：

- 改动摘要。
- 涉及文件列表。
- 自动检查输出。
- 手工验收步骤和结果。
- 未完成项或 `needs_review` 列表。

不得用"页面能打开"替代"信息化闭环已完成"。

### 前端与业务双重验收原则

本项目验收分为两条线：

| 角色 | 主要验收内容 |
|---|---|
| Codex | 代码结构、数据契约、保存回填、公式实现、仓库卫生、自动化检查 |
| 用户 | 前端页面观感、表格复刻准确性、业务文案、字段是否多填/漏填、结论是否符合原始记录 |

Codex 侧自动化检查必须同时覆盖静态准入和运行时准入：`scripts/audit_business_readiness.py`
负责 YAML/MD 业务风险，`scripts/audit_runtime_acceptance.py` 负责 16 产品结构 API、页面路由、
字段 key、保存入口和别嘌醇链接泄漏。

Agent 每次交付页面或产品时，必须提供一个"用户可直接操作"的验收入口：

```text
访问 URL：
测试批号：
建议抽查产品/阶段/检测项：
建议输入样例：
预期页面表现：
预期计算结果：
预期结论：
对应 MD 位置：
```

用户前端验收时重点看：

1. 页面是否能从批次列表进入目标产品，而不是跳回别嘌醇。
2. 产品名、阶段名、检测项编号、检测项顺序是否和 `schema/md_canonical/<品种>.md` 一致。
3. 表格行列、合并单元格、标题层级、字段位置是否和原始记录一致。
4. 操作文案、标准规定、清场条目、异常处理、附件/图谱提示是否和 MD 一致。
5. 日期字段是否是可选择日期，且保存后刷新不被今天覆盖。
6. 批号、检验者/日期、复核者/日期等是否只需填一次，并能在相关页面自动复用。
7. 手填字段是否都能输入；自动计算字段是否只读。
8. 输入样例数据后，自动计算结果是否符合业务预期。
9. 结论中的数值、单位、百分号、符合/不符合格式是否和原始记录一致。
10. 刷新页面或切换检测项后，已填数据是否仍在。
11. 不需要的内容是否没有被擅自添加，例如用户明确不要的平均值、RD 或空白列。
12. 移动到其他产品/阶段后，别嘌醇专属文案、常数、对照品、波长不应泄漏过去。

用户抽查建议：

- 每次至少抽查 1 个中间体检测项、1 个待包装品检测项、1 个成品检测项。
- 对复杂产品额外抽查含量、溶出度/释放度、有关物质、微生物限度。
- 对每个抽查项至少做一次"输入 -> 自动计算 -> 保存 -> 刷新回填 -> 结论检查"。
- 如发现与 MD 或业务不一致，应记录为验收问题，不让 Agent 直接用推断修正，除非用户明确确认。
