## 2. 财务合规风险

### 2.1 中国税务与发票合规

| 风险描述 | 详细说明 | 缓解措施 |
|----------|----------|----------|
| 金税系统无法直连 | ERPNext 没有内置中国金税/税控接口，发票开具和认证仍需外部系统 | 保留现有发票流程，ERPNext 仅做账簿记录；评估第三方金税接口或 RPA 方案 |
| 数电发票不支持 | ERPNext 无内置数电发票功能 | 同上用友或第三方发票系统处理，ERPNext 记录发票台账 |
| 税务申报表无法自动生成 | ERPNext 不能生成中国税务申报表 | 从 ERPNext 导出数据，在现有申报系统或 Excel 中生成申报表 |
| 审计不接受 ERPNext 输出 | 审计师可能不熟悉 ERPNext，对导出格式提出质疑 | POC 阶段邀请审计师参与，提前确认报表格式和审计轨迹可被接受 |

**合规原则**：
- ERPNext 在 POC 和并行期**不替代**税务申报和发票系统。
- 用友继续作为税务基准系统，ERPNext 先验证做账能力。
- 只有财务、审计、税务三方确认后，才考虑逐步替代。

### 2.2 财务报表一致性

| 风险描述 | 详细说明 | 缓解措施 |
|----------|----------|----------|
| 科目映射错误 | Workspace 现有 `FinanceStatementAccountMapping` 与 ERPNext 科目体系不一致 | 建立映射对照表，双重验证 |
| 重分类逻辑差异 | Workspace 的重分类引擎与 ERPNext 标准逻辑不同 | Workspace 重分类引擎保留，基于 ERPNext snapshot 数据计算 |
| 往来重分类缺失 | ERPNext 无内置往来重分类功能 | Workspace 补充重分类计算层 |
| 现金流量表编制差异 | ERPNext 现金流逻辑与中国习惯不同 | Workspace 基于 ERPNext 数据重新生成中国式现金流量表 |

## 3. 数据迁移风险

### 3.1 历史数据策略

| 数据类型 | 迁移策略 | 风险 |
|----------|----------|------|
| 科目表 | 导出用友/Workspace 科目表，导入 ERPNext | 低：科目表结构简单，可校验 |
| 期初余额 | 按公司+年度导入 | 低：需确保借贷平衡 |
| 历史凭证 | 选择性导入（如最近 1-2 年） | 中：数据量大，需批量导入工具 |
| 历史报表 | **不迁移**，保留在 Workspace | 低：Workspace 作为历史数据归档 |
| 客户/供应商 | 导入主数据 | 低：需去重和清洗 |
| 库存数据 | 重新盘点后录入 ERPNext | 中：需物理盘点与系统数据对齐 |

### 3.2 数据迁移原则

1. **不追求全部历史数据迁移**：财务数据保留在 Workspace 作为历史归档，ERPNext 从新期间开始。
2. **双系统并行期**：并行运行 1-2 个期间，逐月比对差异。
3. **迁移前备份**：Workspace 数据库全量备份，ERPNext 实例定期快照。

## 4. 权限与审计风险

### 4.1 ERPNext 权限模型

| 维度 | Workspace 现有 | ERPNext |
|------|---------------|---------|
| 权限体系 | RBAC（resource-based） | RBAC（role + permission） |
| 粒度 | 资源级（finance.ledger.access/write/delete） | 文档类型级（DocType + 操作） |
| 审批流 | 自定义工作流 | 内置 Workflow |
| 审计日志 | 应用层记录 | 内置 Audit Trail |

**风险**：两套权限模型不一致，用户需要适应新的权限体系。
**缓解**：
- Workspace 管入口和展示，ERPNext 管 ERP 内部操作。
- 建立角色映射表（Workspace role → ERPNext role）。
- 关键操作（写回 ERPNext）必须走 Workspace 审批流。

### 4.2 审计轨迹

| 需求 | ERPNext 支持度 |  gaps |
|------|----------------|-------|
| 凭证修改记录 | ✅ Audit Trail | 需确认是否满足中国审计要求 |
| 电子签名 | ❌ 无内置 | 需评估是否必须，或可用审批流替代 |
| 数据不可篡改 | ⚠️ 数据库管理员可修改 | 需通过权限和流程控制 |
| 报表导出追溯 | ✅ 有 | - |

## 5. 运维与业务连续性风险

### 5.1 自部署 ERPNext 运维

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| 服务器故障 | 自部署环境硬件/网络故障 | 使用云 VPS + 自动备份；准备快速恢复方案 |
| 版本升级失败 | ERPNext 升级导致数据或功能异常 | 升级前在 staging 环境验证；保留回滚快照 |
| 备份失效 | 备份文件损坏或恢复失败 | 定期做恢复演练；保留多份备份 |
| 安全漏洞 | ERPNext 或依赖组件存在 CVE | 及时跟进安全公告；限制公网暴露 |

### 5.2 业务连续性方案

**并行运行期（Phase 3）**：
- 用友继续作为基准系统，ERPNext 故障不影响日常做账。
- ERPNext 仅用于验证和数据同步，不产生唯一事实数据。

**切换后**：
- 制定 ERPNext 灾难恢复计划（RTO < 4 小时，RPO < 1 小时）。
- 保留用友只读访问，用于历史查询和应急。

## 6. 人员与变更管理风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 财务人员抵触新系统 | 拒绝使用 ERPNext，继续使用用友/Excel | 让财务人员参与 POC；并行期证明系统可靠性；提供充分培训 |
| 操作人员培训不足 | 录入错误、流程混乱 | 制定培训计划；先培训核心用户，再扩散 |
| IT 团队维护能力不足 | ERPNext 故障无法及时修复 | 培训 IT 团队；建立与 Frappe 社区/合作伙伴的支持渠道 |

## 7. 回滚方案

### 7.1 各阶段回滚策略

| 阶段 | 回滚触发条件 | 回滚动作 | 数据影响 |
|------|-------------|----------|----------|
| Phase 1 POC | POC 结果不达标 | 停止 ERPNext，继续用友 + Workspace | 无，仅测试数据 |
| Phase 2 只读对接 | API 不稳定/数据不一致 | 关闭 sync，Workspace 切回 local 数据源 | 无，snapshot 可删除 |
| Phase 3 并行运行 | 差异过大/财务不认可 | 停止 ERPNext 录入，恢复用友为唯一系统 | ERPNext 数据保留但不再更新 |
| Phase 4 切换后 | 重大问题（税务/审计不通过） | 紧急恢复用友为基准，ERPNext 降级为辅助 | 需补录用友期间数据 |

### 7.2 回滚检查清单

- [ ] 用友系统始终保持可运行状态（并行期）
- [ ] Workspace 始终支持 `sourceSystem = local` 回退
- [ ] 定期备份 ERPNext 数据库和附件
- [ ] 保留用友到 ERPNext 的科目映射文档
- [ ] 明确回滚决策人和审批流程

## 8. 验收标准

每项高风险项必须满足以下验收标准才能进入下一阶段：

| 风险项 | 验收标准 |
|--------|----------|
| 财务报表一致性 | 与用友同期间报表差异 < 0.01% 或差异可解释 |
| 税务合规 | 财务负责人书面确认税务处理方案可行 |
| 审计合规 | 审计师确认 ERPNext 输出可被审计接受 |
| 数据迁移 | 迁移数据 100% 校验通过，无丢失 |
| 权限安全 | 渗透测试通过，关键操作有审计日志 |
| 业务连续性 | 灾难恢复演练通过，RTO/RPO 达标 |
| 人员就绪 | 核心用户培训完成，操作考核通过 |
