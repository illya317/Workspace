# ERP 选型与 Workspace 转型管理层汇报稿

> 对应框架 Agent A：10 页以内汇报结构、一页结论、一页成本收益、一页风险和 POC 里程碑。

---

## 一页结论

**问题**：Workspace 已自研了完整的总账引擎（科目/凭证/余额/重分类/报表映射/底稿校对）和基础库存模块，正走在重复建设 ERP 核心的路上。继续扩展将面临专业性风险、维护成本失控和数据一致性风险。

**方案**：**ERPNext 是当前最值得优先验证的开源 ERP 底座**。先用 ERPNext 做本地 POC，不立即替换用友；Workspace 不再继续扩展 ERP 核心，转为 ERPNext 的上层分析、AI、报表校对和统一入口。

**决策原则**：
1. 用友在 POC 和并行期继续作为财务基准系统。
2. ERPNext 先验证事实能力，不先做大规模二开。
3. Workspace 停止新增完整采购、库存、销售、应收应付和总账引擎。

**投入**：POC 阶段以内部 DIY 为主，基础设施成本与现状持平（¥2,000-5,000/年），主要成本是人力时间。3 年 TCO 保守估计与现状持平，但可避免自研 ERP 核心的长期巨额投入。

**下一步**：启动 Phase 1 ERPNext 本地 POC，选择 1 家最简单的公司，4 周内完成财务数据导入和报表比对。

## 一页中国本地化评估

| 维度 | 现状 | 说明 |
|------|------|------|
| 中国会计准则科目表 | ✅ 可适配 | ERPNext 科目表完全可自定义导入；社区 `erpnext_china` 应用提供预设中国科目表和税务模板 |
| 总账/凭证/报表 | ✅ 完整 | ERPNext 标准财务模块支持 |
| 增值税处理 | ⚠️ 可配置 | Tax Template 机制可配置进项/销项/转出；`erpnext_china` 提供基础模板 |
| 金税系统对接 | ❌ 无内置 | `erpnext_china` 也未覆盖；需外部衔接 |
| 数电发票 | ❌ 无内置 | 社区方案尚未成熟；需外部衔接 |
| 发票查验/出口退税 | ❌ 无 | 需外部系统或自研接口 |

**结论**：ERPNext + `erpnext_china` 可以覆盖**基础本地化**（科目表、税务模板、语言），但**核心税务功能（金税对接、数电发票、发票查验、出口退税）仍需外部衔接**。这与 SAP、Oracle 在中国的现状一致——国际 ERP 都需要本地化插件和税务外挂。

## 一页风险与 POC 里程碑

### 主要风险

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 中国税务/发票/金税无法原生支持 | 🔴 高 | 保留现有税务流程，ERPNext 先做账簿 |
| 财务报表与用友不一致 | 🔴 高 | 并行跑账 1-2 个期间，逐月比对 |
| 继续自研 ERP 核心导致成本失控 | 🔴 高 | **立即停止扩展，转向 ERPNext** |
| 数据迁移错误 | 🟠 中 | 旧数据保留在 Workspace，新数据走 ERPNext |
| 财务人员不接受 | 🟠 中 | 让财务参与 POC，证明可靠性后再推广 |
| ERPNext 自部署运维 | 🟠 中 | 云 VPS + 自动备份，IT 团队培训 |

### POC 里程碑

| 阶段 | 时间 | 交付物 | 通过标准 |
|------|------|--------|----------|
| **Phase 0** | 当前 | 管理层汇报、对比材料、POC 计划 | 管理层批准启动 POC |
| **Phase 1** | 4-6 周 | 单公司财务数据导入、报表比对 | 与用友差异 < 0.01% 或可解释 |
| **Phase 2** | 4-6 周 | 多公司/多币种/成本中心验证 | 财务负责人书面确认 |
| **Phase 3** | 2-3 个月 | 1 家公司并行运行 | 每月差异可控，无阻断性问题 |
| **Phase 4** | 视结果 | 决策是否替代用友 | 审计/税务/财务三方确认 |

**回滚策略**：每个阶段都有明确回滚条件，Workspace 始终支持切回 `sourceSystem = local`，用友始终保持可运行状态。

## 1. 当前问题

### 1.1 Workspace 正在重复建设 ERP

基于代码盘点（`workspace-erpnext-impact-analysis.md`），Workspace 已自研：

- **完整总账引擎**：科目表、会计期间、凭证、余额滚动计算、重分类规则与结果、报表映射、底稿校对。
- **基础库存模块**：原料/包材/成品主数据、批次、出入库流水。
- **成本分析层**：发货、销售工资、成本结构、车间日报（Excel 导入后分析）。
- **预算模块**：部门预算、研发预算（Excel 导入）。

这些正是 ERP 的核心能力。继续扩展意味着长期维护一个自研 ERP，风险包括：
- **专业性风险**：税务、审计、多准则兼容超出内部团队专业范围。
- **维护成本**：总账引擎的持续迭代、bug 修复、新准则适配。
- **数据一致性**：多系统（用友、Excel、Workspace）数据不同步，人工对账成本高。
- **人员依赖**：核心开发人员离职后系统维护困难。

### 1.2 多系统并存痛点

| 痛点 | 现状 | 影响 |
|------|------|------|
| 用友做账 → Excel 汇总 → Workspace 展示 | 数据链条断裂，每次更新需人工搬运 | 信息滞后、易出错 |
| 库存数据在 Excel / 局部系统中 | 没有统一库存台账 | 库存不准、采购/生产决策困难 |
| 销售/发货/回款分散在不同表格 | 无法实时追踪客户欠款 | 回款管理粗放 |
| 财务重分类和报表映射手工维护 | 年度切换时规则迁移工作量大 | 报表产出慢、易出错 |

### 2.1 四方案评分

|----------------|----------|---------|----------------|----------|

| API 与二次开发 | 5 | 4 | 4 | 2 |
| 总拥有成本（3 年） | 1 | 4 | 4 | 3 |
| 升级维护风险 | 1 | 3 | 3 | 4 |
| 用户体验和培训 | 4 | 3 | 3 | 4 |
| **综合** | **2.7** | **3.7** | **3.1** | **3.7** |

### 2.2 各方案结论

| 方案 | 结论 |
|------|------|
| **继续自研 Workspace ERP** | **不推荐**。已投入大量工程成本，但继续扩展的长期风险和成本不可控。 |
| **ERPNext** | **首选 POC 对象**。主仓库核心 ERP 模块开源免费，财务/库存/生产/采购一体化。不付费前提下财务落地复杂度更低、完整开源路径更清晰。中国税务/发票需外部衔接，这是国际 ERP 的共同问题。 |
| **Odoo Community** | **备选但不优先**。有基础会计能力（分录/付款/部分对账），但高级报表/资产/预算需 Enterprise/OCA 补齐，不付费前提下财务落地复杂度高。 |
| **继续用友** | **基准系统保留**。财务合规基线稳定，但 API 弱、管理分析弱、信息流通差。 |

## 3. 推荐方案：ERPNext + Workspace 双系统

### 3.1 分工定位

```
ERPNext = 标准 ERP 事实系统
Workspace = 轻量门户 + 权限 + 分析 + 审批/协同 + AI + 本地特殊流程
```

**ERPNext 负责**：
- 会计事实：科目、凭证、GL Entry、财年、发票、付款。
- 业务事实：客户、供应商、物料、仓库、库存流水、采购、销售、生产、固定资产。
- 标准能力：权限、表单、审批、标准报表、审计日志、附件。

**Workspace 负责**：
- 管理层驾驶舱和分析。
- 中国式财务报表展示、底稿、校对、特殊调整。
- AI 查询、自然语言问数、异常解释。
- 工作汇报、工作计划、人事行政轻流程、文档中心、资料库。
- ERPNext 数据缓存、跨系统统一入口和角色友好的轻量页面。

### 3.2 数据策略

```
ERPNext API
  ↓ read-only sync
Workspace Snapshot（带 Erp 前缀，只读缓存）
  ↓ service compute
Workspace Analysis / Review / AI
```

写回 ERPNext 必须单独审批：
```
Workspace Draft → 审批 → ERPNext API write → 记录 externalId
```

## 4. 落地路线

| 阶段 | 时间 | 关键交付 | 决策点 |
|------|------|----------|--------|
| **Phase 0** | 当前-2 周 | 管理层汇报、对比材料、POC 计划、团队分工 | 管理层批准 |
| **Phase 1** | 4-6 周 | 本地 ERPNext 环境、1 家公司财务数据导入、报表比对 | 差异是否可接受 |
| **Phase 2** | 4-6 周 | ERPNext connector、Account/GL Entry/Customer/Supplier snapshot、Workspace 报表切换数据源 | 技术可行性确认 |
| **Phase 3** | 2-3 个月 | 财务和库存并行跑 1-2 个期间、用友保留为基准、每周记录差异 | 是否具备切换条件 |
| **Phase 4** | 视结果 | 决策是否替代用友、制定切换窗口、备份/回滚演练 | 管理层最终决策 |
