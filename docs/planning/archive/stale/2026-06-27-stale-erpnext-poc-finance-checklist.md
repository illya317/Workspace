# ERPNext 财务 POC 可行性分析

> 对应框架 Agent B：评估 ERPNext 财务模块能力、用友替代差距、中国会计/税务/发票适配情况。

## 1. ERPNext 财务模块能力总览

### 1.1 核心会计功能

| 功能 | ERPNext 支持度 | 说明 |
|------|----------------|------|
| 科目表（Chart of Accounts） | ✅ 完整 | 可自定义导入，支持多层级、多公司 |
| 会计期间（Fiscal Year / Period） | ⚠️ 有机制 | 支持财年、期间锁定、结账，需 POC 验证是否符合用友式操作习惯 |
| 凭证（Journal Entry） | ⚠️ 有机制 | 支持附件、审核/反审核、多行分录，需 POC 验证是否符合用友式操作习惯 |
| 总账（General Ledger） | ✅ 完整 | GL Entry 自动过账，支持查询和导出 |
| 明细账 | ✅ 完整 | 按科目/期间/凭证查询 |
| 科目余额表（Trial Balance） | ✅ 完整 | 标准输出 |
| 资产负债表（Balance Sheet） | ✅ 完整 | 标准格式 |
| 利润表（Profit & Loss） | ✅ 完整 | 标准格式 |
| 现金流量表（Cash Flow） | ✅ 基础 | 标准模块有，但格式需适配中国习惯 |
| 多公司（Multi-Company） | ✅ 完整 | 支持多公司账套、公司间交易 |
| 多币种（Multi-Currency） | ✅ 完整 | 支持汇率、外币凭证、汇率损益 |
| 成本中心（Cost Center） | ✅ 完整 | 支持预算和实际对比 |
| 项目（Project） | ✅ 完整 | 支持项目维度核算 |
| 固定资产（Asset） | ✅ 完整 | 支持多种折旧方法 |
| 预算（Budget） | ✅ 基础 | 支持预算编制和差异分析 |

### 1.2 与 Workspace 现有财务引擎的对比

| 能力 | Workspace 现有 | ERPNext | 差距评估 |
|------|---------------|---------|----------|
| 科目层级 | ✅ 多级 | ✅ 多级 | 相当 |
| 凭证录入 | ✅ 完整 | ✅ 完整 | 相当 |
| 余额滚动计算 | ✅ 自研引擎 | ✅ 自动 | ERPNext 更成熟 |
| 重分类系统 | ✅ 完整（规则+结果+余额调整） | ⚠️ 无内置重分类 | **Workspace 领先** |
| 报表映射 | ✅ mapping-based | ⚠️ 标准报表为主 | **Workspace 领先** |
| 底稿/校对 | ✅ workpaper + review | ❌ 无 | **Workspace 领先** |
| 中国式报表展示 | ✅ 自定义 | ⚠️ 需适配 | **Workspace 领先** |
| 导入追溯 | ✅ 完整（批次+指纹+校验） | ⚠️ 基础 | **Workspace 领先** |

**关键结论**：ERPNext 在标准总账引擎上比 Workspace 自研更成熟，但在**重分类、中国式报表、底稿校对、导入追溯**等管理层分析层功能上，Workspace 已投入大量工程且形成差异化价值。这正是 "ERPNext 存事实，Workspace 做增强" 的分工依据。

## 2. 中国本地化评估

### 2.1 中国会计准则科目表

| 评估项 | 结论 | 证据 |
|--------|------|------|
| 中国企业会计准则科目表能否导入 | ⚠️ 可配置/可导入 | ERPNext 科目表完全可自定义，支持 CSV/Excel 导入标准科目编码体系；社区 `erpnext_china` 提供预设模板，但非官方原生预置完整中国财税方案 |
| 一级科目（1xxx-6xxx）兼容性 | ✅ 良好 | 资产/负债/权益/成本/损益大类完全可映射 |
| 辅助核算（部门/项目/往来/数量/外币） | ⚠️ 部分 | ERPNext 支持 Cost Center / Project / Party，但"往来单位"概念需映射为 Customer/Supplier |

### 2.2 中国本地化社区应用 `erpnext_china`

存在社区维护的 `erpnext_china` 应用（Gitee: `yuzelin/erpnext_china`），提供：

| 功能 | 状态 | 说明 |
|------|------|------|
| 中国科目表 | ✅ 有 | 预设中国会计准则科目表 |
| 税务模板 | ✅ 有 | 基础税种模板 |
| 语言本地化 | ✅ 有 | 中文界面优化 |
| 金税系统对接 | ❌ 无 | 社区应用未提供金税/税控接口 |
| 数电发票解析 | ❌ 无 | 不支持 XML/OFD 解析 |
| 发票查验 API | ❌ 无 | 未接入国家税务总局发票查验 |
| 出口退税 | ❌ 无 | 未提供出口退税申报数据准备 |

**关键判断**：`erpnext_china` 解决了**基础本地化**（科目表、税务模板、语言），但**核心税务功能（金税对接、数电发票、发票查验、出口退税）尚未覆盖**。这与印度形成鲜明对比——印度有官方级的 `India Compliance` 应用，完整覆盖 GST 电子发票、TDS 等税务对接。

### 2.3 增值税处理

| 评估项 | 结论 | 说明 |
|--------|------|------|
| 增值税三级科目（2171 应交税费） | ⚠️ 可配置 | ERPNext 有 Tax Template 机制，可配置进项/销项/转出科目；配合 `erpnext_china` 可加快配置 |
| 进项税额抵扣 | ⚠️ 可配置 | 需自定义 Tax Template 和规则 |
| 销项税额计算 | ⚠️ 可配置 | 同上 |
| 金税系统直连 | ❌ 无内置 | ERPNext 没有内置中国金税/税控设备接口，`erpnext_china` 也未提供 |
| 数电发票 | ❌ 无内置 | 需通过第三方接口或自研 app 对接；社区方案尚未达到生产可用 |

### 2.4 发票与税务申报

| 评估项 | 结论 | 说明 |
|--------|------|------|
| 发票管理 | ⚠️ 基础 | ERPNext 有 Sales Invoice / Purchase Invoice，但无中国发票版式 |
| 金税四期对接 | ❌ 无 | 需外部系统或自研集成 |
| 税务申报表生成 | ❌ 无 | 需从 ERPNext 导出数据后在外部系统申报 |
| 汇算清缴支持 | ❌ 无 | 需人工处理或外部工具 |

### 2.5 审计与合规

| 评估项 | 结论 | 说明 |
|--------|------|------|
| 审计日志 | ✅ 有 | Audit Trail 记录用户操作 |
| 电子签名 | ❌ 无内置 | 需自定义开发或第三方模块 |
| 报表导出审计 | ✅ 有 | Print Format 支持 PDF 导出 |
| 数据备份恢复 | ✅ 有 | 支持数据库备份、实例导出 |

### 2.6 中国本地化外部事实

- **中国税务环境**：数电发票已推广，金税系统要求 ERP 能与税务链路衔接。ERP 本地化通常需要外部流程衔接，而非内置全部税务功能。
- **中国会计准则（CAS）**：与 IFRS 高度趋同，但税务处理差异大（增值税、企业所得税）。
- **ERP 本地化通常需要**：金税/发票/税务申报外部流程衔接，而非 ERP 内置全部税务功能。
- **对比印度**：ERPNext 在印度有成熟的 `India Compliance` 官方应用，完整覆盖 GST 电子发票、TDS 等。中国本地化生态成熟度远低于印度。

**本地化结论**：
- ERPNext + `erpnext_china` **可以覆盖基础本地化**（科目表、税务模板、语言）。
- **不能直接替代**中国税务、发票、金税全部要求——金税对接、数电发票、发票查验、出口退税等核心税务功能仍需外部衔接。
- 科目表和总账层面**可以适配**中国会计准则。
- 税务/发票层面**必须做外部衔接**（第三方工具、RPA、或自研接口）。
- 这不是 ERPNext 独有的问题——SAP、Oracle 在中国同样需要本地化插件和税务外挂。

## 3. 用友替代验证 POC 清单

### 3.1 数据导入验证（必须）

| 步骤 | 验证内容 | 验收标准 |
|------|----------|----------|
| 1 | 导入 1 家公司完整科目表 | 科目编码、名称、层级与用友一致 |
| 2 | 导入期初余额 | 期初借贷平衡，与用友期初一致 |
| 3 | 导入 1 年凭证 | 凭证数量、借贷金额、摘要与用友一致 |
| 4 | 运行余额计算 | 逐月余额与用友同期间结果逐项比对，差异 < 0.01 |
| 5 | 输出标准报表 | 总账、明细账、科目余额表、资产负债表、利润表 |
| 6 | 与用友报表比对 | 所有报表科目金额与用友一致或差异可解释 |

### 3.2 功能验证（必须）

| 功能 | 验证要点 | 风险等级 |
|------|----------|----------|
| 凭证字号 | 是否支持按期间自动编号、是否支持手动调整 | 低 |
| 附件管理 | 凭证是否支持上传附件、附件是否可追溯 | 低 |
| 审核/反审核 | 有审核/反审核机制，需验证是否符合用友式操作习惯 | 中 |
| 结账/期间锁定 | 有期间锁定机制，需验证是否符合用友式操作习惯 | 中 |
| 多公司 | 多公司科目是否独立、合并报表是否可行 | 中 |
| 多币种 | 外币凭证、汇率、汇兑损益 | 中 |
| 成本中心 | 成本中心维度核算、预算对比 | 低 |
| 项目核算 | 项目维度收支统计 | 低 |

### 3.3 中国特殊需求验证（必须）

| 需求 | 验证方法 | 预期结果 |
|------|----------|----------|
| 增值税科目处理 | 配置 Tax Template，模拟进项/销项业务 | 税额计算正确，科目分录符合中国习惯 |
| 固定资产折旧 | 录入资产，运行折旧 | 折旧金额与用友一致 |
| 往来重分类 | 年底应收贷方/应付借方重分类 | 需 Workspace 重分类逻辑补充 |
| 现金流量表 | 运行标准现金流报表 | 格式需调整为中国式，数据需校验 |

### 3.4 验收输出

1. **用友 vs ERPNext 差异表**：逐科目、逐期间比对，记录差异金额和原因。
2. **差异解释和修复方案**：对于差异项，说明是数据问题、配置问题还是系统限制。
3. **财务负责人书面确认**：是否认可 ERPNext 输出结果，是否同意替代用友做账。
4. **不可替代项清单**：明确列出 ERPNext 无法满足、必须用友或外部系统补充的事项。

## 4. 关键风险与假设

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 中国税务/发票无法原生支持 | 高 | 确认税务申报仍走现有流程或用友，ERPNext 只做账簿 |
| 现金流量表格式不符 | 中 | Workspace 基于 ERPNext snapshot 重新生成中国式现金流 |
| 往来重分类逻辑差异 | 中 | Workspace 重分类引擎保留，基于 ERPNext 数据计算 |
| 财务人员接受度低 | 中 | 并行跑账 1-2 个期间，对比验证后再决策 |
| 历史数据迁移复杂 | 中 | 旧数据保留在 Workspace，新数据走 ERPNext |
| 审计要求不满足 | 高 | 提前与审计沟通，确认 ERPNext 输出可被审计接受 |

## 5. 建议的 POC 范围

### 第一阶段（2-4 周）：单公司财务数据验证

- 选择 1 家业务最简单的公司
- 导入科目表 + 期初余额 + 1 个季度凭证
- 输出报表并与用友比对
- 评估差异可解释性

### 第二阶段（2-4 周）：扩展验证

- 增加 1-2 家公司
- 验证多公司、多币种、成本中心
- 测试固定资产、预算模块
- 财务负责人参与评审

### 第三阶段（2-4 周）：并行运行

- 1 家公司同时在用友和 ERPNext 做账
- 每月比对差异
- 记录问题清单和解决方案
